    The API as an exploration contract.

    MEMORA's REST API is not a thin CRUD surface — it is designed as a relational exploration interface backed by stable design principles.

    5 Core resources
    REST Django REST Framework
    Public Read access, no auth required

    Core principle

    The API is a contract

    A stable contract between backend and frontend enables independent evolution of each layer without breaking integrations.

    • Independent evolution of the Next.js frontend
    • Domain stability enforced at the backend level
    • Predictable integration for scripts and pipelines
    • Consistency and clarity over ad-hoc endpoint growth

    Resource-oriented design

    5 domain resources

    Each entity in the domain model is exposed as a REST resource with list and detail endpoints.

    🧵 pieces
    ✦ designers
    ◷ eras
    ◈ categories
    ⌂ maisons

    Resources are represented as JSON, keeping the domain model explicit and predictable for both humans and tooling.

    Relational retrieval

    First-class relational context

    Exploration requires relational context. The API supports retrieving entities together with their key relationships — a Piece with its Designer, Era, and Category in a single request.

    This makes relationships directly usable at the client level, without multiple round-trips to reconstruct context.

    ORM optimisation

    The backend applies select_related and prefetch_related as the dataset grows to keep response times manageable.

    Exploration mechanism

    explanation›domain-model.md

    Fashion archives as a relational knowledge system.

    MEMORA models archival records as a network of interconnected entities — where relationships between objects are the primary structure of exploration, not metadata.

    5 Domain entities
    4 Core relationships
    ∞ Exploration paths

    Core design principle

    Relationships are navigation, not metadata

    In many archival systems, relationships are treated as descriptive attributes. In MEMORA, relationships define the structure through which users explore the archive.

    • A Piece links to its Designer, Era, Category, and Maison
    • Each link is a navigational entry point — not a label
    • Exploration flows in multiple directions across the graph
    • The archive becomes traversable, not just searchable

    Central entity

    Piece as entry point

    A Piece is not only an object with attributes. It is an entry point into a network of related entities — each enabling further exploration.

    PieceCentral entity
    →
    DesignerAuthorship
    →
    MaisonInstitution
    PieceCentral entity
    →
    EraPeriod
    +
    CategoryTypology
    ✦

    Designer

    Creator of Pieces. Enables exploration by authorship and links Pieces to institutional context through a Maison.

    ◷

    Era

    Historical fashion period. Provides temporal structure and enables navigation across chronological context.

    ◈

    Category

    Typological classification. Enables structural exploration across garment types and formal relationships between Pieces.

    ⌂

    Maison

    Institutional fashion house. Provides organisational context and connects Designers and Pieces to broader histories.

    Relational structure

    Multi-directional navigation

    The domain model is a graph of explicit foreign key relationships enabling traversal in both directions.

    Piece → Designer from object to authorship
    Designer → Pieces from creator to full body of work
    Piece → Era from object to historical period
    Era → Pieces from period to all its objects
    Designer → Maison from creator to institution
    Piece → Category from object to typology

    Domain concept

    Completeness as a progressive quality signal

    MEMORA treats completeness as a progressive quality signal rather than a strict validation requirement.

    Archival records often evolve incrementally. Early records may contain partial information and be enriched over time.

    Progressive enrichment

    Incomplete records can exist and be explored. Completeness is a filterable dimension, not a gate. Records are enriched over time without blocking usability.

    Alignment with architecture

    The architecture serves the domain model

    The domain model directly informs the system architecture at every layer. It is not an artefact of implementation — it is the conceptual foundation.

    Backend
    API exposes domain entities preserving their relational structure via DRF serializers
    Frontend
    Next.js UI uses relational structure to enable filtering, navigation, and exploration workflows
    Database
    Relational schema with foreign keys mirrors the domain model exactly — no impedance mismatch

    Rationale

    Why a relational model

    Fashion objects exist within interconnected systems of authorship, history, and classification. A relational model provides:

    • Structural clarity
    • Referential integrity
    • Efficient relational querying
    • Scalable exploration capabilities

    Document-based storage would make relational exploration more complex and less explicit.

    Summary

    The MEMORA domain model treats fashion archives as a relational knowledge system. By modelling explicit entities and relationships, the system enables exploration, analysis, and creative discovery. The domain model forms the conceptual and structural foundation of the entire system.

    Common to all entities

    Base mixin fields

    id UUIDModel
    UUID v4 — random, non-sequential. Primary key on all entities. Prevents enumeration and supports distributed imports.
    slug SlugModel
    URL-safe identifier. Auto-generated from name or title on first save. Stable — never regenerated. Conflicts get numeric suffix: yohji-yamamoto--1.
    created_at TimeStampedModel
    Timezone-aware UTC datetime. Set once on creation, never modified. Indexed.
    updated_at TimeStampedModel
    Updated on every .save() call. Not triggered by .update() queryset calls. Not indexed.

    Filtering — relational & attribute-based.

    Query string filters for relational navigation, text search, and completeness detection. Powered by django-filter.

    3 Relational filters
    5 Completeness fields

    Piece filters — relational

    Filter by designer, era, or category

    Relational filters accept the slug of the related entity. Each filter narrows results to pieces linked to that specific entity.

    designer GET /api/pieces/?designer=pierre-balmain
    Returns all pieces whose designer.slug matches the given value.
    era GET /api/pieces/?era=1950s
    Returns all pieces whose era.slug matches the given value.
    category GET /api/pieces/?category=dress
    Returns all pieces whose category.slug matches the given value.

    Implementation

    Filter pattern

    Filters are passed as query string parameters. Multiple filters can be combined in a single request.

    /api/pieces/
  ?designer=yohji-yamamoto
  &era=annees-80
  &category=coat
    Backend
    Powered by django-filter — declarative filter classes per endpoint.
    Filter values
    Relational filters use slugs, not IDs. Slugs are stable and URL-safe.

    Piece filters — text

    Search by title

    The title_contains filter performs a substring match against the piece title.

    GET /api/pieces/?title_contains=dress

    Returns all published pieces whose title contains the given string. Can be combined with relational filters to narrow results further.

    Piece filters — completeness

    Complete pieces only

    The is_complete=true filter returns only pieces that satisfy all 5 completeness criteria.

    GET /api/pieces/?is_complete=true
    • title
    • description
    • image
    • category
    • materials_data

    Permissions — read-public, write-restricted.

    Read operations are open to all users. Write operations are restricted to admin users. Enforced via Django REST Framework permission classes.

    Public Read access — no authentication required
    Admin Write access — authenticated admin only
    DRF Django REST Framework permission classes

    Permission model

    Access by HTTP method

    Public — no auth required
    GET HEAD OPTIONS
    Admin only — authentication required
    POST PATCH PUT DELETE

    Read access

    Public — all users

    All GET endpoints are publicly accessible. No authentication header is required to browse, list, or retrieve any resource.

    List
    GET /api/pieces/
    Paginated list of published pieces
    Detail
    GET /api/pieces/{id}/
    Single piece by UUID or slug
    Designers
    GET /api/designers/
    All designers, with pieces
    Related
    GET /api/maisons/
    Also: categories, eras

    Write access

    Admin only

    All write operations — create, update, and delete — require an authenticated admin user. Unauthenticated write requests are rejected.

    • POST — create a new resource
    • PATCH — partial update
    • PUT — full update
    • DELETE — remove a resource
    Unauthenticated response
    403 Forbidden

    Implementation

    Django REST Framework

    Permissions are enforced declaratively using DRF permission classes applied at the ViewSet level.

    Read endpoints
    IsAuthenticatedOrReadOnly
    Safe methods allowed for all. Unsafe methods require auth.
    Write endpoints
    IsAdminUser
    Only users with is_staff=True can write.
    Summary
    Read-public
    All GET endpoints are publicly accessible — no authentication required to browse the archive
    Write-restricted
    POST, PATCH, PUT, DELETE are restricted to authenticated admin users. Rejected requests receive 403 Forbidden
    DRF enforcement
    Permissions are declared at ViewSet level using DRF permission classes — no middleware or manual checks needed

    Your first exploration.

    Onboarding tutorial — from zero to navigating the MEMORA archive through relational context.

    By the end of this tutorial

    What you will be able to do

    • Explore archival Pieces from the frontend interface
    • Navigate relational context — Designer, Era, Category
    • Understand how relationships enable discovery across the archive
    Prerequisite

    MEMORA must be running locally before starting this tutorial. See how-to/run-the-project.md if needed.

    Step 1

    Open the exploration interface

    Open your browser and navigate to the MEMORA frontend, then go directly to the pieces list.

    Frontend
    http://localhost:3000/
    Pieces list
    http://localhost:3000/pieces
    Each Piece is a fashion object connected to a broader relational context.

    Step 2

    Open a Piece

    Click any Piece in the list. You are now viewing the Piece detail page. Observe the fields displayed.

    Descriptive
    titledescription
    Relational
    designereracategory

    Relational fields are not just metadata — they are entry points into the archive.

    Step 3

    Explore relational context

    Designer, Era, and Category are relationships — each one connects this Piece to other Pieces in the archive.

    Designer Connects to other Pieces created by the same person — exploration by authorship.
    Era Connects to Pieces from the same historical period — exploration by time.
    Category Connects to Pieces of the same garment type — exploration by typology.

    Step 4

    Relational exploration

    Instead of browsing a fixed hierarchy, MEMORA allows exploration through relationships.

    From a single Piece, you can reach
    • Other Pieces by the same Designer
    • Other Pieces from the same Era
    • Other Pieces in the same Category

    A Piece is an entry point into a network — not an isolated record.

    Step 5

    Completeness is progressive

    Some Pieces may have more information than others — this is expected. Archival data evolves over time.

    • Incomplete records can exist in the archive
    • Records are enriched progressively
    • Exploration remains possible at every stage

    Step 6 — optional

    Understand the backend structure

    The frontend retrieves all data from the MEMORA REST API. You can query it directly to see the raw relational model.
